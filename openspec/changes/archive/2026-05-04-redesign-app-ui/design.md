## Context

Art Frame 是一款数字画框应用，支持从本地文件、媒体库、WebDAV、SMB 等多种图源加载图片并以幻灯片形式播放。当前 UI 基于标准 Material 3 卡片式布局，功能完整但视觉表现力与「数字画框」的产品定位存在落差：

- **图源页面**：纵向列表 + 小缩略图（132×92），信息密度高但缺乏视觉冲击力
- **播放页面**：全屏黑底 + 固定可见的控件层，缺少沉浸感
- **设置页面**：标准卡片分组，功能性足够但风格与画廊定位不匹配
- **导航**：始终可见的 AppBar + 底部/侧边导航，占据大量屏幕空间

目标是将 UI 从「功能工具」升级为「沉浸式画廊体验」，同时保持极简现代的视觉语言。

## Goals / Non-Goals

**Goals:**
- 将图源页面改造为沉浸式网格画廊，以大尺寸图片预览为核心视觉元素
- 播放页面实现控件自动隐藏，提供纯净的全屏观赏体验
- 建立统一的极简设计系统：深色优先、大留白、精致动效
- 简化导航 chrome，让内容占据最大空间
- 所有改动仅限表现层，不改变数据模型和业务逻辑

**Non-Goals:**
- 不改变现有的路由结构（仍为 sources / settings / playback 三个目的地）
- 不改变状态管理方案（Riverpod）
- 不添加新的业务功能（如收藏、分类、搜索等）
- 不改变图源的数据模型和持久化方式
- 不做自定义主题/换肤功能

## Decisions

### 1. 图源页面：从列表改为瀑布流/网格画廊

**选择**：采用自适应网格布局（非瀑布流），每个图源以大尺寸封面图为主体，标题和操作以半透明叠层覆盖在图片底部。

**理由**：
- 瀑布流虽然视觉丰富，但图源数量通常不多（3-10个），瀑布流在少量项目时显得空洞
- 等高网格更整齐，在少量图源时也能保持视觉平衡
- 叠层信息减少视觉噪音，让图片成为主角

**替代方案**：
- 保持列表但放大缩略图 → 仍然是传统列表感，不够画廊
- 全屏轮播式 → 一次只能看一个图源，管理效率低

### 2. 播放页面：控件自动隐藏 + 手势操作

**选择**：控件（返回按钮、计数器、导航箭头、标题）在用户触摸/鼠标移动后显示，3秒无操作后自动淡出。支持左右滑动切换图片。

**理由**：
- 数字画框的核心场景是「挂在墙上静静展示」，控件应该在不需要时完全消失
- 手势操作比按钮更自然，减少视觉元素

**替代方案**：
- 始终显示半透明控件 → 破坏沉浸感
- 仅手势无按钮 → 可发现性差，新用户不知道如何操作

### 3. 图片切换动效：交叉淡入淡出升级

**选择**：将当前 180ms 的简单 crossfade 升级为 600ms 的带缩放的交叉淡入淡出（旧图微缩 + 淡出，新图微放大 + 淡入），营造「呼吸感」。

**理由**：
- 当前 180ms 过快，缺乏优雅感
- Ken Burns 效果（持续缓慢缩放平移）在静态展示时很经典，但实现复杂且可能分散注意力
- 带缩放的 crossfade 是平衡优雅感和实现复杂度的最佳选择

### 4. 导航简化：隐藏式底部导航

**选择**：
- 手机端：底部导航栏改为更紧凑的样式，减小高度，使用图标+微型标签
- 平板/桌面端：侧边导航改为可收起的图标栏（类似 macOS 侧边栏），默认收起仅显示图标
- 移除 AppBar，页面标题融入内容区域

**理由**：
- AppBar 在画廊应用中是多余的 chrome，标题可以融入页面内容
- 紧凑导航释放更多空间给内容

### 5. 主题系统：深色优先 + 中性色调

**选择**：
- 保持 Material 3 但大幅调整色彩：降低种子色饱和度，偏向中性灰紫
- 深色模式作为默认推荐（画框场景更适合深色背景）
- 减小圆角半径（28→16 for cards, 18→12 for buttons），更现代锐利
- 统一动效曲线为 `Curves.easeOutCubic`，时长 300-600ms

**替代方案**：
- 完全自定义主题不用 Material 3 → 工作量大，且失去 Material 组件的无障碍支持
- 保持当前高饱和紫色 → 与画廊的中性优雅定位不符

### 6. 动画方案：使用 Flutter 内置动画

**选择**：使用 Flutter 内置的 `AnimatedOpacity`、`AnimatedScale`、`AnimatedSlide`、`SlideTransition` 等组件，不引入第三方动画库。

**理由**：
- 本次需要的动效（淡入淡出、缩放、滑动）都是基础动画，Flutter 内置完全够用
- 避免引入额外依赖，保持项目轻量
- `flutter_animate` 等库的声明式 API 虽然方便，但对于有限的动效需求来说是过度引入

## Risks / Trade-offs

- **[风险] 网格布局在少量图源时可能显得空旷** → 设计优雅的空状态和「添加图源」占位卡片填充空间
- **[风险] 控件自动隐藏降低可发现性** → 首次进入播放页时显示控件并延迟隐藏（5秒），之后恢复3秒
- **[风险] 图片叠层文字在浅色图片上不可读** → 使用渐变遮罩（底部到顶部的黑色渐变）确保文字始终可读
- **[权衡] 移除 AppBar 后页面缺少统一的顶部锚点** → 通过页面内的标题区域和一致的间距系统补偿
- **[权衡] 减小圆角可能让部分用户觉得不够「友好」** → 16px 仍然是柔和的圆角，只是比 28px 更现代
- **[权衡] 深色优先可能在户外强光下可读性降低** → 保留亮色模式选项，仅改变默认推荐
