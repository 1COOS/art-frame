## Context

当前 `art-frame` 已有三类可播放来源：内置 `bundled`、本地离散文件 `localFiles`、本地目录 `localDirectory`。这些能力已经复用统一的 `MediaSource / MediaItem / localSourcesController / selectedSourceController / PlaybackPage` 主链路，说明 Sources -> Playback 的骨架已经足够承载新的图源类型。

Phase 3 的问题不再是“如何播放图片”，而是“如何把系统媒体库中的图片资产接入现有图源模型”。与本地文件和目录不同，媒体库资产通常依赖平台权限、稳定资产标识与按需解析；在 Android / iOS / macOS 上也存在不同权限与文件可达性语义。因此本阶段需要在保持最小闭环的同时，为后续更复杂的媒体资产能力留出扩展空间。当前实现阶段仅承诺 Android / iOS 入口与闭环，macOS 虽在长期规划中属于可评估平台，但本阶段暂不开放媒体库导入入口。


结合现有实现与插件能力，本阶段更适合采用“先选择一批图片资产，再将其保存为一个静态媒体库图源”的策略，而不是一开始就做系统相册浏览器、增量同步或动态监听。`photo_manager` 一类插件可提供：
- 访问系统媒体库权限
- 图片资产枚举与选择所需标识
- 通过稳定 asset id 在重启后重新解析资产
- 在支持平台上获取图片文件或展示数据

## Goals / Non-Goals

**Goals:**
- 在 Android / iOS 提供系统相册 / 媒体库图片导入入口，并创建新的媒体库图源。
- 持久化所选媒体资产的稳定标识与最小展示元数据，支持应用重启后恢复。
- 让媒体库图源复用既有 Sources 列表、选中态与 Playback 闭环。
- 定义最小权限处理与失败语义，使未授权、无可用资产、资产失效等情况有明确行为。
- 为后续相册分组浏览、增量刷新与更复杂资产类型保留扩展空间。

**Non-Goals:**
- 不实现视频、Live Photo、GIF 特殊播放语义。
- 不实现按系统相册分组浏览、智能相册、搜索或筛选。
- 不实现媒体库变更监听、自动刷新或增量同步。
- 不设计复杂权限恢复策略、后台预热或离线缓存数据库。
- 不改变当前 Playback 的核心轮播模型与应用壳层导航结构。

## Decisions

### 1. Phase 3 采用“媒体库选择后生成静态图源”而不是内建完整媒体库浏览器
- 决策：本阶段只承诺提供媒体库图片选择入口，让用户选择一批图片资产并生成一个可复用图源；不在应用内实现完整系统相册浏览层。
- 原因：这与 Phase 1/2 的“先打通最小可用闭环”策略一致，也能最大限度复用现有 Sources 页交互，而不把工作量转移到新页面和复杂 UI。
- 备选方案：
  - 在应用内实现完整媒体库网格/相册浏览：功能更强，但会显著扩大 Phase 3 范围。
  - 仅请求权限并导入“最近项目”：实现更简单，但会失去用户选择控制，不符合图源创建心智。

### 2. 媒体库图源持久化“稳定资产标识 + 最小展示元数据”，而不是仅保存瞬时文件路径
- 决策：媒体库图源中的条目应保存可跨重启重新解析的 asset id，以及标题/描述/排序所需的最小元数据；运行时再将 asset 解析为可显示图片。
- 原因：系统媒体库中的底层文件路径并不总是稳定或可长期直接访问，仅保存瞬时文件路径会让恢复语义脆弱；而稳定 asset id 更符合媒体库模型。
- 备选方案：
  - 只保存解析后的文件路径：实现简单，但恢复可靠性弱。
  - 导入时复制原图到应用私有目录：恢复最稳，但会引入大文件复制、磁盘管理与隐私复杂度，超出本阶段目标。

### 3. Playback 继续复用现有单一播放页，但需要为媒体库条目补齐解析分支
- 决策：Playback 不新增独立页面；媒体库图源仍通过既有 selectedSource + item 列表驱动，只是在渲染层支持解析媒体资产图片。
- 原因：当前 Playback 已支持基于 item 列表的轮播与切换，新增图源类型不应破坏这条统一链路。
- 备选方案：
  - 为媒体库图源单独做专用播放页：会造成播放逻辑分叉。
  - 在导入时把媒体资产全部转换为本地文件副本：可继续沿用 file 渲染，但代价过大。

### 4. Phase 3 将支持平台限定在能稳定访问系统媒体库的目标平台
- 决策：Phase 3 首版能力只在 Android / iOS 承诺；macOS 暂不开放入口，Web 与其他不适配平台隐藏或禁用入口。
- 原因：媒体库访问强依赖平台权限与系统 API。虽然插件在 macOS 上存在基础能力，但当前 `art-frame` Phase 3 仅完成 Android / iOS 的最小闭环，若过早承诺 macOS 会制造已支持但入口未开放的歧义。
- 备选方案：
  - 所有平台统一展示入口：会制造伪支持体验。
  - 仅先做移动端：范围最小，也与当前实现一致；macOS 留待后续单独评估入口、权限和插件行为后再开放。

### 5. 权限不足和资产失效都按“导入失败 / 恢复失败但不崩溃”处理
- 决策：当媒体库权限不可用、未选中图片、恢复时资产已失效或不可再访问时，系统应返回空结果或失效条目处理，不创建脏图源，也不让 Playback 崩溃。
- 原因：媒体库资产受系统权限和用户外部操作影响更强，失败语义需要在 spec 里先定清楚。
- 备选方案：
  - 将失效资产静默保留并在播放时报错：用户感知差。
  - 自动删除整个图源：过于激进，最好保留为后续产品决策。

## Risks / Trade-offs

- [平台权限差异导致实现分叉] → 通过统一媒体库访问服务抽象权限请求、资产解析与支持平台判断，减少页面层分支。
- [资产标识恢复成功但文件解析失败] → 在恢复阶段过滤失效条目，并为空图源提供明确空态或不导入语义。
- [媒体库条目过多导致导入性能下降] → Phase 3 只处理“用户已选中的一批资产”，不做全库扫描。
- [现有 `MediaItem` 仅有 `asset/file` 语义，难以直接表达媒体库资产] → 在实现中允许扩展 item/source 模型或增加平台解析层，但不把范围扩展为通用媒体引擎重构。
- [不同平台的 limited permission 语义不同] → Phase 3 只要求最小可用行为：有访问则可选图，无访问则不创建图源。

## Migration Plan

1. 引入媒体库访问依赖并封装统一服务，先验证权限请求、图片选择与 asset id 读取可行。
2. 扩展图源 / 条目建模与持久化结构，使媒体库图源可保存稳定资产标识与最小元数据。
3. 在 Sources 页增加媒体库导入入口，并在导入成功后复用既有 selectedSource / Playback 跳转闭环。
4. 为 Playback 增加媒体库条目解析与展示分支，同时补齐删除、恢复与失效处理。
5. 通过自动化测试与至少一个目标平台手工验收收口；若出现严重回归，可先回退媒体库入口与解析分支，不影响既有 bundled/localFiles/localDirectory 闭环。

## Open Questions

- Phase 3 是否需要在 UI 上把媒体库图源标记为独立 badge，例如 `Media library`，还是先沿用更中性的本地图源表达？
- 恢复时若一个媒体库图源只剩部分资产可访问，是否保留剩余条目，还是把整个图源视为失效？
- 第一版是直接使用系统 picker 返回选择结果，还是使用插件能力先拉取最近图片再在应用内做最小选择 UI？
