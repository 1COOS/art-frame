## Context

当前的 Flutter 应用已经完成一轮 Material 3 统一，但 Sources 和 Playback 这两个高频入口仍然承载了偏多的常驻文案、徽标、元信息与按钮。Sources 列表已具备导入、选择、删除、进入播放、远程缩略图渲染等完整链路，但列表项默认同时展示描述、多个 metadata badge、状态 chip 和多组动作，扫描成本偏高。Playback 已经是独立沉浸式页面并支持远程图片渲染，但顶部来源标签、底部信息块和上一张/下一张按钮持续占据视觉注意力，弱化了图片本身。

另外，当前 source 编辑能力并不存在于列表页，需要基于既有 network source 对话框和 source 持久化结构扩展，而本地文件、目录、媒体库与网络图源的可编辑范围并不完全相同，因此需要在简化 UI 的同时明确哪些类型支持编辑、哪些仍保持只读概览。

## Goals / Non-Goals

**Goals:**
- 让 Sources 列表默认只呈现有助于选择与识别 source 的关键信息，降低单卡片阅读负担。
- 在 Sources 管理流中加入编辑既有 source 的入口，并复用现有图源配置能力而不是引入全新维护入口。
- 让 Playback 页面减少常驻辅助信息，优先突出图片内容，仅在必要时暴露导航与切换控制。
- 保持既有导入、删除、选中、播放、远程鉴权渲染与路由进入方式不变，把改动限制在表现层和最小必要的 source 更新路径。

**Non-Goals:**
- 不重做 source domain model、导入流程或 remote media fetching 机制。
- 不新增新的 source 类型或改变 source 持久化格式。
- 不引入复杂的播放手势、缩放、批量编辑或 source 排序功能。
- 不把本次变更扩展到 settings、theme token 或其他页面的二次重构。

## Decisions

### 1. Sources 列表采用“摘要优先，细节后置”的卡片策略
- 保留缩略图、标题、source kind、必要状态与最少量摘要信息（如图片数量），移除默认常驻的大段描述和过多 metadata badge。
- 将 source 维护动作收敛为更轻量的操作区，优先保证“选择 / 播放 / 编辑 / 删除”可发现，而不是让静态说明占据空间。
- 备选方案：继续保留现有多 badge 方案，仅靠样式弱化信息。未选原因是信息条目数量本身已经过多，只改颜色和层级不足以显著降低噪音。

### 2. 编辑能力优先覆盖可稳定重建配置的图源类型
- 网络图源已有 `NetworkSourceDraft`、验证对话框和目录浏览对话框，可复用其配置与浏览流程来完成编辑闭环，因此优先支持网络图源编辑。
- 对本地目录、媒体库和本地文件图源，若缺少稳定的可逆编辑语义，则保留只读卡片与既有删除/重新导入路径，避免为了“统一编辑”引入不可靠的伪编辑能力。
- 备选方案：所有 source 一律开放编辑入口。未选原因是本地图源的来源选择往往依赖系统选择器，硬做编辑会把“重选资源”与“更新元数据”混在一起，增加实现复杂度和认知负担。

### 3. Playback 改为“媒体优先、控制按需”的覆盖层
- 图片继续全屏铺陈，顶部只保留最必要的返回能力；来源名、数量、播放间隔等信息降级为轻量、可收起或次级展示，不再同时作为强视觉元素常驻出现。
- 上一张/下一张控制保留，但视觉权重下降，避免与图片争夺注意力；空态仍保持明确返回 Sources 的路径。
- 备选方案：完全隐藏控制，仅通过手势或键盘操作。未选原因是桌面与触控场景并存，且当前产品仍需要明显可见的基础操作入口。

### 4. 路由 contract 通过修改现有 app-navigation-foundation 规格对齐当前实现
- 现有 `app-navigation-foundation` 规格仍把 Playback 描述为保留 shell UI 的目的地，但当前实现已经改为独立沉浸式页面。
- 本次 change 直接修正该能力定义，使规格与当前产品方向一致，并明确即便继续从 Sources 闭环进入 Playback，也不要求保留共享 shell chrome。
- 备选方案：避免改 spec，只在实现中继续偏离。未选原因是会让 proposal/spec 与现状长期失真，影响后续变更判断。

## Risks / Trade-offs

- [Sources 信息压缩过度] → 通过保留标题、类型、选中态和条目数量，确保用户仍能快速辨认 source；对需要维护的 source 提供编辑入口承接更多信息。
- [仅部分 source 支持编辑导致体验不一致] → 在文案与交互上将编辑定义为“支持的 source 可编辑”，其余类型继续通过删除后重建处理，避免承诺超出现有能力边界。
- [Playback 控制过弱影响可发现性] → 保留明确返回与前后切换按钮，只降低视觉重量，不一次性切到纯手势模式。
- [规格与现有实现不一致] → 在本次 spec delta 中同步修正 app-navigation-foundation，避免后续归档时继续携带错误导航假设。
