## Context

`art-frame` 当前已经拥有统一的本地图源主链路：`MediaSource / MediaItem` 负责图源与条目建模，`localSourcesController / selectedSourceController` 负责持久化与选择态，`Sources -> Playback` 负责导入后的预览闭环。Phase 4 的核心问题不再是“如何播放图片”，而是“如何把多个网络协议来源接入到这条既有主链路里，同时避免为每种协议重复建模与重复 UI”。

网络图源与本地图源的主要差异在于：
- 需要保存连接配置而不是本地文件路径
- 目录浏览与文件枚举往往是异步、远程、可能失败的
- 不同协议在认证方式、路径语义、文件元数据与错误类型上存在差异
- 首阶段并不适合同时做完整协议实现、全量同步、离线缓存与复杂重试机制

因此 Phase 4 更适合先完成一层统一 foundation：定义远程 source 的公共模型、配置结构、验证与浏览契约、最小恢复语义，以及与 Playback 主链路的接缝；随后再将 WebDAV、SMB、SFTP 作为后续协议型 change 按优先级逐步落地。

## Goals / Non-Goals

**Goals:**
- 建立统一的网络图源基础抽象，覆盖远程 source 的协议类型、连接配置、目录或文件入口、最小展示元数据与持久化结构。
- 约束 WebDAV、SMB、SFTP 三类协议共享的用户流程：创建来源、填写配置、验证连接、浏览目录或枚举文件、生成可播放图源、进入 Playback、重启后恢复。
- 明确当前阶段的协议差异边界，包括认证方式、路径格式、目录浏览能力、失败语义与后续拆分方向。
- 让现有 `Sources -> Playback` 主链路可以承载未来网络图源，而不要求立刻在单个 change 中交付全部协议实现。
- 为后续独立协议 change 留出稳定 capability 边界，避免未来反复重写 domain / controller / persistence 结构。

**Non-Goals:**
- 不在本 proposal 阶段承诺一次性交付 WebDAV、SMB、SFTP 的完整生产级实现。
- 不实现离线缓存数据库、缩略图预热、断点传输、后台同步与增量刷新。
- 不实现复杂认证体系（如 OAuth、Kerberos、密钥代理、多因素校验）的完整产品方案。
- 不引入独立远程媒体浏览器页面体系；仍以复用现有 Sources / Playback 壳层为原则。
- 不改变当前 Playback 的核心轮播模型，只定义远程图片条目进入这条链路的契约。

## Decisions

### 1. Phase 4 先做统一 foundation，再拆分协议实现
- 决策：本 change 只建立统一网络图源基础抽象与最小闭环要求，不要求在一个 change 中把 WebDAV、SMB、SFTP 全部实现完成。
- 原因：三类协议在实现细节上差异较大，若一开始混在一个 change 中推进，容易把 proposal / design / implementation 范围膨胀到不可控。
- 备选方案：
  - 为每种协议直接单独建模：实现会更直观，但公共能力会在 domain、controller、持久化与 UI 层反复复制。
  - 先只做单协议（如 WebDAV）：范围最小，但会让统一抽象滞后，后续协议接入成本升高。

### 2. 网络图源需要独立于现有 local/mediaLibrary 的来源配置模型
- 决策：网络图源应有明确的远程来源配置模型，至少包含协议类型、主机 / 路径或共享名、认证字段、根目录或目标路径、可选显示名称，以及与 `MediaSource` 关联的稳定 source id。
- 原因：本地目录和媒体库的持久化结构不足以表达远程连接配置，若继续复用现有字段会导致语义混乱、协议差异被硬塞进 path 字段。
- 备选方案：
  - 将远程配置直接塞进 `MediaSource.description/path`：实现快，但会污染现有模型。
  - 每种协议各自定义一套完全独立 source：可表达差异，但会破坏统一 Sources 主链路。

### 3. 统一“连接验证 -> 浏览 / 枚举 -> 生成静态图源”的交互心智
- 决策：用户流程采用统一的三步心智：配置远程来源、验证连接、浏览目录或枚举文件后生成一个静态可播放图源。
- 原因：这与 Phase 1/2/3 的“导入后生成可复用 source”模式一致，也能减少未来协议间 UI 分叉。
- 备选方案：
  - 让 Playback 直接绑定实时远程目录：实现上更接近文件系统，但恢复、错误处理与缓存复杂度过高。
  - 每次进入播放都重新浏览远程目录：用户心智不稳定，也不利于最小恢复闭环。

### 4. 首阶段持久化“来源配置 + 条目快照”，而不是完整远程同步状态
- 决策：持久化层先保存来源配置与导入时选择出的媒体条目快照；恢复时优先保证 source 能回到 Sources 列表，并在需要时重新解析远程内容。
- 原因：数字相框场景下最重要的是“图源配置不丢”和“重启后可继续播放或至少可重新进入”，而不是第一版就做完整远程同步状态机。
- 备选方案：
  - 保存完整远程目录树快照：恢复更完整，但复杂度和存储开销过高。
  - 完全不保存条目快照、每次现拉：恢复依赖网络可达，体验不稳定。

### 5. app-navigation-foundation 只做最小扩展，不引入新导航骨架
- 决策：继续复用现有 shell 与 Playback destination，只在 spec 层扩展“网络图源导入后可进入预览”的行为要求。
- 原因：远程 source 是图源类型扩展，而不是新的顶层产品模式。
- 备选方案：
  - 单独做远程来源 wizard / browser 子导航：未来可能需要，但首阶段不应提前复杂化。

## Risks / Trade-offs

- [统一抽象过早，压扁协议差异] → 在 foundation 里只定义公共契约，把协议特有能力放到后续独立 change。
- [远程恢复依赖网络，重启体验不稳定] → 首阶段要求最小恢复语义：至少 source 配置与条目快照可恢复，不强制立即完成远程刷新。
- [认证字段设计不足导致后续协议扩展困难] → 配置模型按“公共字段 + 协议扩展字段”组织，避免一次性锁死。
- [目录浏览和文件枚举差异过大] → 统一 spec 里只要求“浏览或枚举之一可成立”，具体由协议型 change 细化。
- [Sources 页面交互过载] → 首阶段只定义统一入口与最小创建流，不设计复杂远程管理台。

## Migration Plan

1. 新增 `network-source-foundation` 与 `remote-protocol-sources` specs，明确统一远程图源能力与协议差异边界。
2. 对 `app-navigation-foundation` 增加网络图源导入后进入 Playback 的 requirement 更新。
3. 后续按优先级拆分独立实现 change（推荐先 WebDAV，再 SMB，最后 SFTP），每个 change 复用 foundation 模型与主链路。
4. 当首个协议实现落地后，再评估是否需要补充缓存、认证升级、缩略图与同步机制。
5. 如果某个协议实现风险过高，可单独推迟，不影响 foundation 与其他协议推进。

## Open Questions

- 首个协议实现是否确定以 WebDAV 作为优先落地点，还是需要先做更细的产品比较？
- 网络图源条目在恢复后若连接失败，是保留快照播放能力、展示错误态，还是直接要求用户重新验证？
- 用户创建网络图源时，是允许“目录型图源”和“离散文件型图源”并存，还是先统一成目录型入口？
- 是否需要在 foundation 阶段就定义敏感信息（密码 / 密钥）的存储策略，例如 keychain / keystore，而不是普通持久化？
